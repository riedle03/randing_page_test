How We Built This

이 페이지는 원래 문서에 있던 실제 작업 과정을 빠뜨리지 않으면서, 현재 랜딩 페이지와 같은 디자인 언어로 다시 편집한 버전입니다. 한 화면에 모든 정보를 몰아넣지 않고, 스크롤하면서 단계별로 따라갈 수 있게 구성했습니다.

Overview

실제 작업 흐름 전체 보기

이 문서는 https://dhs-fabulous-site-09d196.webflow.io/ 클론을 AI와 협업해서 만든 실제 과정을 그대로 정리한 가이드입니다. 교사와 학습자가 "이런 방식으로 웹을 만들 수 있구나"를 체감할 수 있도록 구성했습니다.

전체 흐름 한눈에 보기
  • 사이트 분석 → 무엇을 만들지 이해
  • PRD 작성 → 어떻게 만들지 문서화
  • 첫 번째 빌드 → 일단 만들기
  • 비교 & 피드백 → 차이점 발견
  • PRD 수정 → 이해를 업데이트
  • 두 번째 빌드 → 더 정확하게 다시 만들기
  • 부분 수정 → 디테일 다듬기
핵심 원칙: 코드를 먼저 쓰지 않는다. 이해를 먼저 문서화한다.
이 페이지의 역할

읽는 문서와 보는 화면 사이 어딘가에 있는, 스크롤형 빌드 노트입니다.

01
Step 1. Analyze

사이트를 먼저 분석했습니다.

클론할 사이트를 AI에게 분석시켰습니다. 감상 수준이 아니라, 구조와 수치와 레이어 관계가 바로 구현으로 이어질 수 있도록 해체하는 단계였습니다.

사용한 프롬프트 
https://dhs-fabulous-site-09d196.webflow.io/ 사이트를 분석해줘.
섹션 구조, 색상, 폰트, 레이아웃, 컴포넌트, 애니메이션 방식을 상세히 알려줘.
HTML 구조와 CSS 클래스도 가능하면 포함해줘.
AI가 발견한 것들
발견 내용눈으로 볼 때AI 분석 후
h1 크기"크다"63px, font-weight: 600
카드 색상"은색"#E5E0DA (티타늄 실버)
카드 크기"신용카드처럼 생겼다"480×300px, rx:12.874px
폰트"Apple 같은 느낌"system-ui, -apple-system
레이아웃"섹션이 있다"모든 요소 `position: fixed`, `.stage` 4000px
애니메이션"스크롤하면 뭔가 나온다"Webflow IX2 scroll-driven
배움 포인트

AI는 "느낌"을 숫자와 코드로 바꿔준다. 디자이너가 감각적으로 파악한 것을 개발 언어로 번역하는 역할을 한다.

02
Step 2. PRD

가장 중요한 건 PRD였습니다.

Product Requirements Document는 만들기 전에 "무엇을 어떻게 만들 것인지"를 정의하는 문서입니다. 여기서 기준을 세워두어야 이후 생성과 수정이 흔들리지 않습니다.

사용한 프롬프트
분석 결과를 바탕으로 PRD를 만들어줘.
동일한 스펙 문서로, 이걸 바탕으로 만드는 게 나을 것 같아.
PRD에 들어간 내용
  • 제품 개요, 구현 범위
  • 레이아웃 구조, DOM 계층, z-index
  • 애니메이션 시퀀스
  • 컴포넌트 스펙, 색상 팔레트, 타이포그래피
  • 반응형 스펙, 에셋 목록, 기술 비교
  • 구현 우선순위와 검수 체크리스트
PRD 없이 만들면
  • AI가 매번 다른 해석을 한다
  • 수정할 때 기준이 없다
  • 무엇이 맞고 틀린지 알 수 없다
PRD가 있으면
  • "PRD대로 만들어줘" 한 줄로 시작 가능
  • 어느 부분이 잘못됐는지 파악 가능
  • 검수 기준이 명확해진다
03
Step 3. First Build

첫 번째 빌드는 빠르게 만들었습니다.

PRD를 기준으로 구현하고, 에셋은 같은 폴더에 있는 실제 파일을 사용했습니다. 핵심은 다시 설명하지 않고 문서를 기준으로 바로 생성에 들어가는 것이었습니다.

사용한 프롬프트 
PRD.md를 읽고 그 스펙 그대로 구현해줘.

에셋은 같은 폴더에 있는 파일 사용:
- gradient-bg.svg
- card-branding.svg
- iphone-p-1080.jpeg

결과물: index.html 단일 파일
조건: 외부 라이브러리 없음, Vanilla JS
핵심 포인트
  • 스펙을 다시 말하지 않는다
  • 에셋 파일을 명시한다
  • 기술 제약 조건을 명시한다
결과

작동하는 페이지가 만들어졌다. 완벽하지 않았지만 기반이 생겼다.

04
Step 4. Compare

핵심은 차이점의 언어화였습니다.

단순히 "이상하다"라고 말하는 것이 아니라, 어떤 장면이 어떤 순서와 관계로 다르게 느껴지는지를 사건 단위로 설명하는 단계였습니다.

나쁜 피드백
좀 이상해, 고쳐줘
좋은 피드백
원래 레퍼런스 사이트는:
- 그라디언트가 크게 있다가 스크롤을 내리면서 점차 축소되고
- 어느 시점에 카드 모양으로 변환되었다가
- 뒤에 폰 목업이 등장하면서 폰 안에 카드가 쏙 들어가는 구조

index.html은:
- 스크롤을 내리면 카드 목업이 등장하고
- 그리고 휴대폰 목업이 등장해서 이질적임
체크리스트
  • 처음 로드 화면이 같은가?
  • 첫 스크롤 이벤트가 같은가?
  • 등장 순서가 같은가?
  • fade, slide, morph 중 무엇인가?
  • 요소들이 어떤 관계로 움직이는가?
  • 마지막 상태가 같은가?
05
Step 5. Update the PRD

이해가 깊어지면 문서도 바뀝니다.

피드백을 통해 애니메이션의 본질을 새롭게 이해했고, 그에 따라 PRD도 업데이트했습니다. 문서는 한 번 쓰고 끝나는 결과물이 아니라, 이해의 최신 상태를 기록하는 도구였습니다.

수정 전
scroll ~20%: 카드 등장
scroll ~50%: 헤딩 등장
scroll ~70%: iPhone 등장
수정 후
핵심 구조: "그래디언트 → 카드 → 폰 진입" 모핑

scroll 0~45%: gradient-big이 scale(3.5→1) + border-radius(34→13px) 모핑
scroll 45~55%: 카드 브랜딩 fade-in
scroll 50~65%: 헤딩 fade-in
scroll 60~80%: iPhone 슬라이드업
scroll 70~90%: 카드가 iPhone 스크린 안으로 이동 + 축소
의미

PRD는 현재 가장 정확한 이해의 기록이다. 만들면서 이해가 바뀌면 문서도 바뀌어야 한다.

06
Step 6. Refine the Technique

결국 기법까지 바꾸게 됐습니다.

새로운 이해를 바탕으로 완전히 다른 구현 기법을 적용했습니다. 이 단계에서 핵심 전환은 transform에서 clip-path로의 이동이었습니다.

처음 시도
gradientMorph.style.transform =
  `translate(${dx}px, ${dy}px) scale(${scale})`;
올바른 방향
gradientMorph.style.clipPath =
  `inset(${top}px ${right}px ${bottom}px ${left}px round ${radius}px)`;
왜 clip-path인가?
  • 요소 자체는 화면 전체를 차지한 채로 고정
  • 보이는 영역만 점점 좁아짐
  • transform-origin 문제 없음
  • 위치 계산이 직관적이다
배움 포인트

실패는 낭비가 아니라 이해를 깊게 만드는 과정이었다.

07
Step 7. Final Tuning

마지막엔 문제 구간만 정확히 다듬었습니다.

전체를 다시 만드는 대신, 어색한 마지막 스크롤 구간만 정밀하게 수정했습니다. 기준 문서가 있으니 어떤 부분만 손봐야 하는지도 명확했습니다.

사용한 프롬프트
마지막 스크롤이 어색해.
그 전까지는 아주 자연스러웠어.
그 부분만 마저 검토하고 수정해줘.
부분 수정이 가능한 이유
  • 어느 단계가 문제인지 특정 가능하다
  • 다른 부분은 건드리지 않아도 된다
  • 수정 후 검수 기준이 명확하다
결론

완성도는 전체를 갈아엎는 데서가 아니라, 마지막 어색한 10~20%를 얼마나 정확히 짚는지에서 결정됐다.

Philosophy

이 방식의 전체 철학

코드를 쓰는 것보다 먼저, 무엇을 만들고 왜 그렇게 보이는지를 이해하는 것이 중요했습니다. 이 프로젝트는 그 철학을 실제로 적용한 기록입니다.

코드보다 이해가 먼저다
나쁜 방식: 일단 코드를 만들고 → 뭔가 이상하면 → 다시 만들기
좋은 방식: 이해를 문서화하고 → 문서 기반으로 만들고 → 차이를 발견하면 → 문서를 업데이트하고 → 다시 만들기
대화의 3가지 유형
유형목적예시
분석 요청이해하기"이 사이트가 어떻게 만들어졌는지 분석해줘"
생성 요청만들기"PRD대로 index.html을 만들어줘"
수정 요청개선하기"마지막 부분이 어색해, 그 부분만 수정해줘"
좋은 프롬프트의 핵심
  • 숫자를 쓴다
  • 관계를 설명한다
  • 기준 문서를 참조한다

Repeatable Workflow

이 방식은 다른 사이트에도 그대로 적용할 수 있습니다.

마음에 드는 사이트를 고르고, AI에게 분석을 요청하고, PRD를 만들고, 구현하고, 비교하고, 문서를 업데이트하고, 다시 빌드하면 됩니다. 이 과정에서 디자이너가 해야 할 일은 사이트를 세밀하게 관찰하고, 차이를 언어로 정확히 표현하고, 어느 수준에서 완성인지 판단하는 것입니다.

이 문서는 실제 작업 과정을 기록한 것입니다. 2026-03-23
