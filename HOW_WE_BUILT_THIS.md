# AI와 함께 랜딩 페이지 만들기

## 이 프로젝트의 실제 작업 과정 전체 기록

이 문서는 `https://dhs-fabulous-site-09d196.webflow.io/` 클론을 AI(Claude Code)와 협업해서 만든 실제 과정을 그대로 정리한 가이드입니다. 교사와 학습자가 "이런 방식으로 웹을 만들 수 있구나"를 체감할 수 있도록 작성했습니다.

## 전체 흐름 한눈에 보기

1. 사이트 분석 → 무엇을 만들지 이해
2. PRD 작성 → 어떻게 만들지 문서화
3. 첫 번째 빌드 → 일단 만들기
4. 비교 & 피드백 → 차이점 발견
5. PRD 수정 → 이해를 업데이트
6. 두 번째 빌드 → 더 정확하게 다시 만들기
7. 부분 수정 → 디테일 다듬기

핵심 원칙: 코드를 먼저 쓰지 않는다. 이해를 먼저 문서화한다.

## STEP 1. 사이트 분석

### 무엇을 했나?

클론할 사이트를 AI에게 분석시켰다.

### 사용한 프롬프트

```text
https://dhs-fabulous-site-09d196.webflow.io/ 사이트를 분석해줘.
섹션 구조, 색상, 폰트, 레이아웃, 컴포넌트, 애니메이션 방식을 상세히 알려줘.
HTML 구조와 CSS 클래스도 가능하면 포함해줘.
```

### AI가 발견한 것들

눈으로만 봐서는 알 수 없는 것들을 AI가 소스 파싱으로 추출했다.

| 발견 내용 | 눈으로 볼 때 | AI 분석 후 |
| --- | --- | --- |
| h1 크기 | "크다" | 63px, font-weight: 600 |
| 카드 색상 | "은색" | #E5E0DA (티타늄 실버) |
| 카드 크기 | "신용카드처럼 생겼다" | 480×300px, rx:12.874px |
| 폰트 | "Apple 같은 느낌" | system-ui, -apple-system (외부 폰트 없음) |
| 레이아웃 | "섹션이 있다" | 모든 요소 `position: fixed`, `.stage` `height: 4000px` |
| 애니메이션 | "스크롤하면 뭔가 나온다" | Webflow IX2 scroll-driven |

배움 포인트: AI는 "느낌"을 숫자와 코드로 바꿔준다. 디자이너가 감각적으로 파악한 것을 개발 언어로 번역하는 역할을 한다.

## STEP 2. PRD 작성

### PRD란?

Product Requirements Document. 만들기 전에 "무엇을 어떻게 만들 것인지"를 정의하는 문서다.

### 사용한 프롬프트

```text
분석 결과를 바탕으로 PRD를 만들어줘.
동일한 스펙 문서로, 이걸 바탕으로 만드는 게 나을 것 같아.
```

### PRD에 들어간 내용

1. 제품 개요: 무엇을 만드는가, 구현 범위
2. 레이아웃 구조: DOM 계층, z-index 표
3. 애니메이션 시퀀스: scroll % → 요소 → 상태 변화 표
4. 컴포넌트 스펙: 각 요소의 px 단위 치수
5. 색상 팔레트: CSS 변수 토큰
6. 타이포그래피: font-size, weight, family
7. 반응형 스펙: 브레이크포인트별 변경사항
8. 에셋 목록: 파일명, 크기, 대체 방법
9. 기술 스펙: 원본 vs 클론 구현 비교
10. 구현 우선순위: Phase 1/2/3 체크리스트
11. 검수 체크리스트: 완성 기준

### PRD가 필요한 이유

PRD 없이 바로 만들면:

- AI가 매번 다른 해석을 한다
- 수정할 때 기준이 없다
- 무엇이 맞고 틀린지 알 수 없다

PRD가 있으면:

- AI에게 "PRD대로 만들어줘" 한 줄로 시작 가능
- 수정 시 PRD의 어느 부분이 잘못됐는지 파악
- 검수 기준이 명확해진다

배움 포인트: 프롬프트 하나로 모든 것을 말하려 하지 말고, 먼저 스펙 문서를 만든 다음 그것을 참조시켜라.

## STEP 3. 첫 번째 빌드

### 사용한 프롬프트

```text
PRD.md를 읽고 그 스펙 그대로 구현해줘.

에셋은 같은 폴더에 있는 파일 사용:
- gradient-bg.svg
- card-branding.svg
- iphone-p-1080.jpeg

결과물: index.html 단일 파일
조건: 외부 라이브러리 없음, Vanilla JS
```

### 핵심 포인트

- 스펙을 다시 말하지 않는다 → "PRD.md를 읽어" 한 줄로 해결
- 폴더에 있는 에셋 파일을 명시 → AI가 실제 파일을 활용
- 기술 제약 조건을 명시 → "라이브러리 없음, 단일 파일"

### 결과

작동하는 페이지가 만들어졌다. 완벽하지 않았지만 기반이 생겼다.

## STEP 4. 비교 & 피드백

### 원본을 보고 차이를 찾는 방법

단순히 "이상해"라고 말하는 게 아니라, 구체적으로 무엇이 다른지 설명한다.

나쁜 피드백:

```text
좀 이상해, 고쳐줘
```

좋은 피드백:

```text
원래 레퍼런스 사이트는:
- 그라디언트가 크게 있다가 스크롤을 내리면서 점차 축소되고
- 어느 시점에 카드 모양으로 변환되었다가
- 뒤에 폰 목업이 등장하면서 폰 안에 카드가 쏙 들어가는 구조

index.html은:
- 스크롤을 내리면 카드 목업이 등장하고
- 그리고 휴대폰 목업이 등장해서 이질적임
```

### 차이점을 발견하는 체크리스트

- 처음 로드했을 때 화면이 같은가?
- 스크롤의 첫 번째 이벤트가 같은가?
- 각 요소가 등장하는 순서가 같은가?
- 요소가 어떻게 등장하는가? fade, slide, morph 중 무엇인가?
- 요소들이 서로 어떤 관계로 움직이는가?
- 마지막 상태가 같은가?

배움 포인트: 디자이너의 핵심 능력이 여기서 발휘된다. 코드를 몰라도 "무엇이 다른지"는 눈으로 볼 수 있다. 그 차이를 언어로 정확히 표현하는 것이 바이브코딩의 핵심 스킬이다.

## STEP 5. PRD 수정

### 피드백을 통해 이해가 바뀐 부분

애니메이션의 본질을 새롭게 이해했다.

수정 전 PRD의 애니메이션 섹션:

```text
scroll ~20%: 카드 등장
scroll ~50%: 헤딩 등장
scroll ~70%: iPhone 등장
```

수정 후 PRD의 애니메이션 섹션:

```text
핵심 구조: "그래디언트 → 카드 → 폰 진입" 모핑

scroll 0~45%: gradient-big이 scale(3.5→1) + border-radius(34→13px) 모핑
             화면을 채운 블롭이 카드 형태로 수렴
scroll 45~55%: 카드 브랜딩 fade-in (카드 완성)
scroll 50~65%: 헤딩 fade-in
scroll 60~80%: iPhone 슬라이드업
scroll 70~90%: 카드가 iPhone 스크린 안으로 이동 + 축소
```

### PRD 수정의 의미

PRD는 한 번 쓰고 끝나는 문서가 아니다. 만들면서 이해가 깊어질 때마다 업데이트한다. PRD는 현재 가장 정확한 이해의 기록이다.

## STEP 6. 두 번째 빌드

### 핵심 기법 전환: transform → clip-path

처음 시도:

```javascript
// 요소에 scale 적용 → transform-origin이 화면 중심이라 카드가 엉뚱하게 움직임
gradientMorph.style.transform = `translate(${dx}px, ${dy}px) scale(${scale})`;
```

올바른 방향:

```javascript
// clip-path: inset()으로 보이는 영역을 점점 줄여나감
// 화면 전체 → 카드 크기/위치로 수렴
gradientMorph.style.clipPath =
  `inset(${top}px ${right}px ${bottom}px ${left}px round ${radius}px)`;
```

### 왜 clip-path인가?

- 요소 자체는 화면 전체를 차지한 채로 고정
- 보이는 영역만 점점 좁아짐
- transform-origin 문제 없음
- 위치 계산이 직관적이다

### 기법 발견 과정

완성된 레퍼런스를 보고 "이게 어떻게 구현됐을까?"를 AI와 함께 역추론하면서 올바른 기법을 찾았다.

배움 포인트: 첫 번째 구현이 틀려도 괜찮다. 틀린 결과를 보면서 "왜 이렇게 되는가"를 이해하는 것이 학습이다. 바이브코딩에서 실패는 낭비가 아니라 이해를 깊게 만드는 과정이다.

## STEP 7. 부분 수정

### 마지막 20% 다듬기

전체를 다시 만들 필요 없이, 문제 있는 부분만 정확히 지정해서 수정했다.

사용한 프롬프트:

```text
마지막 스크롤이 어색해.
그 전까지는 아주 자연스러웠어.
그 부분만 마저 검토하고 수정해줘.
```

### 부분 수정이 가능한 이유

- 어느 단계가 문제인지 특정 가능하다
- 다른 부분은 건드리지 않아도 된다
- 수정 후 검수 기준이 명확하다

## 이 방식의 전체 철학

### 코드보다 이해가 먼저다

나쁜 방식:

```text
일단 코드를 만들고 → 뭔가 이상하면 → 다시 만들기
```

좋은 방식:

```text
이해를 문서화하고 → 문서 기반으로 만들고 → 차이를 발견하면 → 문서를 업데이트하고 → 다시 만들기
```

### 대화의 3가지 유형

| 유형 | 목적 | 예시 |
| --- | --- | --- |
| 분석 요청 | 이해하기 | "이 사이트가 어떻게 만들어졌는지 분석해줘" |
| 생성 요청 | 만들기 | "PRD대로 index.html을 만들어줘" |
| 수정 요청 | 개선하기 | "마지막 부분이 어색해, 그 부분만 수정해줘" |

### 프롬프트 품질의 핵심

나쁜 프롬프트:

- "예쁘게 만들어줘"
- "비슷하게 만들어줘"

좋은 프롬프트:

- 숫자를 쓴다
- 관계를 설명한다
- 기준 문서를 참조한다

## 이 방식으로 다른 사이트도 만들 수 있다

아래 순서를 그대로 반복하면 된다.

1. 마음에 드는 사이트를 고른다
2. AI에게 분석을 요청한다
3. 분석 결과를 바탕으로 PRD를 만든다
4. "PRD대로 만들어줘"로 빌드한다
5. 원본과 비교해서 차이를 언어로 표현한다
6. PRD를 수정하고 다시 빌드한다
7. 만족할 때까지 5~6을 반복한다

이 과정에서 코드를 직접 쓰는 시간은 거의 없다. 디자이너가 해야 할 일은 사이트를 세밀하게 관찰하고, 차이를 언어로 정확히 표현하고, 어느 수준에서 완성인지 판단하는 것이다.
